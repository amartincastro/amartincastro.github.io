00 Overview

DCG's product suite has accumulated technology layers over decades. A given customer's experience spans interfaces written in VB6 (still load-bearing in places), Durandal and Knockout (the previous web generation), ASP.NET Core, Blazor, Angular, and, increasingly, React. Every product ships with the same logo, the same name on the same invoice — but historically each stack had its own UI dialect.

The design system fixes what a DCG component should look like. The hard part has always been how to make that true on every stack the company ships on — without sitting a designer behind each engineer.

01 The design system

The system itself was built in Figma over many years. Colors, typography, spacing, component variants, states, charts, and the patterns that hold them together — all documented as the source of truth for any DCG product.

Designers can read it. Product managers can read it. Engineers can read it. But "readable" doesn't translate into a button that ships with the right corner radius on a Blazor page in a 2019 codebase. For that, somebody has to apply judgment — historically, me.

The system gets unusually detailed where it matters most for financial software. Data visualization, for instance, has its own multi-page treatment: chart anatomy, tooltip type sizes, legend ordering and wrapping behavior, and a separate institution / cross-institution palette specifically for data series — distinct from the brand palette. Charts are the most data-dense UI element in DCG products and the most likely place for brand drift if every stack implements them on its own.

02 Encoding the system

The Claude skill is a single markdown file (~17KB) that translates the Figma system into something a model can apply consistently. Brand identity, the full color palette with usage rules, typography, component standards (buttons, inputs, dropdowns, toggles, date pickers, charts), UX principles, and per-stack output rules. When a developer asks Claude for a component, the skill loads and the response comes back already on-brand.

## Color Palette

| Name      | Hex       | Usage                          |
|-----------|-----------|--------------------------------|
| Primary   | #0055b8   | Primary actions, headings      |
| Secondary | #23A6DE   | Secondary accents, 360° mark   |
| Contrast  | #faa41a   | Highlights, cross-institution  |
| Profit    | #3F724B   | Positive values, gains         |
| Loss      | #AE0101   | Negative values, errors        |
| Disabled  | #F2F2F2   | Muted/inactive surfaces        |

**Rules:**
- Never introduce colors outside this palette
- Profit/Loss reserved for financial meaning only
- Exact hex values — no approximation
- WCAG AA contrast required

The skill is the same shape for every other token — typography, button variants, input states, validation colors, chart palettes, the difference between a generic toggle and the amber cross-institution toggle. Designer intent, machine-readable.

03 Stack-agnostic by design

The unlock isn't the palette — it's the per-stack output. Before the skill generates anything, it detects the project's framework from files in the working directory. Then it adapts the component to that framework's conventions — template syntax, data binding, lifecycle, file layout. 

## Step 1: Detect the Tech Stack

Before generating any component, identify the project's tech
stack by examining:
- `package.json` — Angular, Durandal/Knockout, or other JS frameworks
- `*.csproj` or `*.sln` files — Blazor or ASP.NET
- `*.vbp` or `*.vbw` files — VB6
- Existing component files and folder structure

If you cannot determine the stack, ask the developer which
framework they are using before proceeding.

Adapt all component output to match the detected framework's
conventions (templates, data binding syntax, file structure,
lifecycle, etc.).

A VB6 maintainer gets a form with the right corner radius and the right blue. An Angular dev gets the same component as a properly-structured class with the right inputs and outputs. A Blazor dev gets a razor file with the brand-correct CSS. Same design system, three different idioms, no designer in the loop.

04 Outcome
  • 25 engineers use the skill across DCG's product suite — from VB6 maintainers to React engineers on the newest surfaces.
  • 4 product managers use it to mock up new features directly from specifications, without spending the day in Figma. This is the part I didn't expect.
  • One source of truth, applied automatically to every stack DCG ships. The Figma system stays the constitution; the skill is how it actually lands.
05 What changes when PMs can ship components

The engineer adoption story was the expected one. We made design system compliance cheaper than ignoring it, so engineers ship on-brand by default. Good. Predictable.

The product manager adoption was the surprise. Four PMs who'd never touched Figma started using the skill to convert written specs into working component mockups they could attach to tickets. Designer-as-gatekeeper turned into designer-as-substrate: the system I built encodes my judgment well enough that PMs can apply it without me being in the room.

That's the part I'm thinking about for the next version. The Figma system is finite — you can fully describe the rules once and they're done. The set of people who can usefully apply those rules isn't finite. A skill is a force multiplier for everyone downstream of it.
